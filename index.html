CommonJS

CommonJS是一种应用于服务端的规范.
CommonJS规定了每个文件就是一个模块,有自己的作用域.
在一个文件里面定义的变量,函数,类,都是私有的,对其他文件不可见.(每个文件都有自己的模块作用域)
每个模块内部,module 变量代表当前模块.这个变量是一个对象,它的 exports 属性是对外的接口.
加载某个模块,其实是加载该模块的 module.exports 属性.

require

在模块中如果想加载其他模块,使用 require() 方法.
require()可以返回外部模块所输出的API

特点

在CommonJS中所有代码都运行在模块作用域，不会污染全局作用域。
模块可以多次加载，但是只会在第一次加载时运行一次，然后运行结果就被缓存了，以后再加载，就直接读取缓存结果。
要想让模块再次运行，必须清除缓存。
模块加载的顺序，按照其在代码中出现的顺序。
CommonJS是同步加载的,这个特点适合服务端,而不适合浏览器端.

Flex

Flex之前用什么?

  • normal flow
  • float + clear
  • position relative + absolute
  • display inblock-inblock
  • 负 margin

Flex 来了

块级元素垂直排列,行内元素水平排列.
flex 则与方向无关.
flex 空间自动分配,自动对齐.
flex 适用于简单的线性布局,更复杂的布局交给 grid 布局(请期待下个版本)

flex不翻译为弹性布局,因为之前通过百分比的方式的布局已经被成为弹性布局.

Flex 的属性

Flex container 的属性

  • flex-direction 方向
    • row(默认): 主轴为水平方向,起点在左端.
    • row-reverse: 主轴为水平方向,起点在右端.
    • column: 主轴为垂直方向,起点在上沿.
    • column-reverse: 主轴为垂直方向,起点在下沿.
      demo
  • flex-wrap 换行
    • nowrap(默认值):不换行.
    • wrap: 换行,第一行在上方.
    • wrap-reverse: 换行,第一行在下.
      demo
  • flex-flow 上面两个元素的简写
    flex-flow: <flex-direction> || <flex-wrap>;
  • justify-content 主轴方向对齐方式
    • flex-start(默认值): 左对齐.
    • flex-end: 右对齐.
    • center: 居中.
    • space-between: 两端对齐,item之间的间隔都相等
    • space-around: 每个item两侧的间隔相等.所以,项目之间的间隔比项目与边框的间隔大一倍
      demo
  • align-items 侧轴方向对齐方式
    • flex-start: 侧轴起点对齐.
    • flex-end: 侧轴终点对齐.
    • center: 侧轴中点对齐.
    • baseline: item的第一行文字的基线对齐.
    • stretch: 如果item未设置高度或设为auto,将占满整个容器的高度.
      demo
  • align-content 多行/列内容对齐方式
    • 极少用到.

Flex item 的属性

  • flex-grow (剩余空间多时,剩余空间如何分配)定义项目的放大比例.默认为0,值越大分配到的空间越多.
  • flex-shrink(空间不够时,item如何瘦身)定义项目的放大比例,默认为0.值越大,瘦身程度越大.
  • flex-basis(一般不用) 默认占用大小.不写的话就是原始大小
  • flex 上面三个的缩写. flex: none | [ <'flex-grow'> <'flex-shrink'>? || <'flex-basis'> ]. 建议优先使用这个属性.
  • order 顺序 有了这个就可以不用双飞翼了.通过CSS来控制顺序.
  • align-self 允许单个item有与其他item不一样的对齐方式,可覆盖align-items属性.默认为auto,表示继承父元素的align-items属性.如果没有父元素,则等同于stretch.

模块化

为什么要使用模块化？

首先说说为什么需要模块化

小明和小王各自负责实现不同的两个功能.
两个人分开写的代码最终会在一个应用同时上使用.
两个人事先没做什么约定.结果

两个人的代码因为命名冲突,部分全局变量相互覆盖
两个模块都用到了一些相同的功能,但是两人都各自完成,浪费时间.
最终这个应用炸了…..
两人加班到深夜,不断调试,最终才在Deadline前勉强上线.
更惨的是后来的小李,他要负责维护两人离职后留下的代码……

模块化的意义

  • 解决命名冲突.
  • 减少代码耦合性,便于依赖管理.
  • 提高代码复用性,减少开发成本.
    比如模块1,模块2都需要同一个方法来实现,那么就把这个方法抽离出来做成模块3.
    在1,2中调用模块3.
  • 便于使用,想用哪个功能就加载哪个模块.迭代升级添加新功能也较为方便.
  • 方便阅读和维护.

简单的模块化写法

最原始的写法

这种写法中一个函数就是一个模块,使用哪个模块直接调用相应函数.
缺点很明显,看不出各个模块之间有什么关系
一个模块占据一个全局变量名称,变量名冲突依旧可能发生

万一别的代码中在全局环境执行了一句block1.cur = "Boom!".(也可能是第三方注入)
模块内的重要参数被改变了,模块功能还要出错.

模块化的规范

使用模块化的写法有个前提,就是所有的开发人员遵循同一套规则.

Round 参数：Round可以理解为圆满的铺满.为了实现圆满所以会压缩或拉伸

####背景图片的原始起始位置

参数分别表示背景图片是从边框,还是内边距(默认)值,或者是内容区域开始显示.
盒模型分border-box和padding-box
背景图片自然也分
!如果背景不是no-repeat.这个设置回无效,图片从边框开始.

背景图裁剪

####　背景图大小

1、auto：默认值,不改变背景图片的原始高度和宽度；

2、<长度值>：成对出现如200px 50px,将背景图片宽高依次设置为前面两个值,当设置一个值时,将其作为图片宽度值来等比缩放；

3、<百分比>：0％~100％之间的任何值,将背景图片宽高依次设置为所在元素宽高乘以前面百分比得出的数值,当设置一个值时同上；

4、cover：顾名思义为覆盖,即将背景图片等比缩放以填满整个容器；

5、contain：容纳,即将背景图片等比缩放至某一边紧贴容器边缘为止.

多重背景

如果有 size 值,需要紧跟 position 并且用 “/“ 隔开；
如果有多个背景图片,而其他属性只有一个（例如 background-repeat 只有一个）,表明所有背景图片应用该属性值.
background-color 只能设置一个.

动画

animation-name 指定要绑定到选择器的关键帧的名称

animation-duration 动画指定需要多少秒或毫秒完成

animation-timing-function 设置动画将如何完成一个周期

  • linear 动画从头到尾的速度是相同的.
  • ease 默认值. 动画以低速开始,然后加快,在结束前变慢.
  • ease-in 动画以低速开始.
  • ease-out动画以低速结束.
  • ease-in-out 动画以低速开始和结束.
  • cubic-bezier(n,n,n,n) 使用贝兹曲线函数来描述动画过程.
    前两个值和后两个值分别描述了开始时和结束时曲线的斜率(快慢).
    四个值的范围都是0~1. 前五个效果都是贝兹曲线上的特值.

animation-delay 设置动画在启动前的延迟时间.

animation-iteration-count 定义动画的播放次数.

animation-direction 指定是否应该轮流反向播放动画.

animation-fill-mode 规定当动画不播放时(当动画完成时,或当动画有一个延迟未开始播放时),要应用到元素的样式.

animation-play-state 指定动画是否正在运行或已暂停.

    • ondrop 在一个拖动过程中，释放鼠标键时触发此事件.
